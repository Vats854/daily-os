# Second Brain / Daily OS — Design Direction Brief

## Цель продукта

Сделать не «лайф-дашборд», а рабочую систему управления днем, проектами и фокусом: пользователь открывает приложение и понимает, что важно сейчас, что уже принято в работу, что требует ревью, что изменил/предложил AI-оператор.

## Главное продуктовое обещание

> Я открываю приложение и вижу: что сейчас важно, какие обязательства у меня есть, что требует решения, и что ассистент изменил или поставил под сомнение.

## Визуальная цель

Спокойный, плотный, современный PWA-интерфейс в духе Linear / Weeek / Notion minimal / Readwise / Cron, но без прямого копирования. Светлая база, мягкий голубой/сине-зеленый акцент, аккуратные градиенты только как фоновые акценты, а не декоративный шум.

## Чего избегать

- Не копировать референс 1-в-1.
- Не делать «карточный дашборд ради дашборда».
- Не растягивать кнопки и пустые блоки на весь экран.
- Не добавлять fake analytics, прогресс-бары и проценты без реальных данных.
- Не делать огромные эмоциональные панели вместо рабочих объектов.
- Не прятать действия за декоративным UI.

## Базовые продуктовые объекты

- Inbox item — сырая мысль / входящее сообщение.
- Task — действие со статусом: inbox / backlog / this_week / today / done.
- Project — жизненный или рабочий фокус с этапом hero journey.
- Daily plan — план дня с блоками времени, обязательствами и фактами.
- Review item — что нужно подтвердить, отклонить или уточнить.
- AI action log — что ассистент понял, изменил и почему.

## Главный принцип интерфейса

Каждый видимый блок должен отвечать хотя бы на один вопрос:

1. Какое решение он помогает принять?
2. Какое действие можно сделать прямо отсюда?
3. Какие данные за ним стоят?
4. Можно ли было бы показать это как таблицу/список без потери смысла?

Если нет — блок убрать.

## Информационная архитектура MVP

Навигация:

1. Inbox — захват и разбор входящих мыслей.
2. Сегодня — что делать сейчас, план дня, фактический ход дня.
3. Неделя — фокусы недели, обязательства, переносы.
4. Проекты — большие жизненные/рабочие фокусы и hero journey.
5. Доска — задачи по статусам.
6. Лог — компактный audit trail действий ассистента.

## Hero Journey / геймификация

Геймификация применяется только к большим фокусам и жизненным проектам, не к каждой мелкой задаче.

Этапы:

- call — Замысел
- commitment — Решение
- preparation — Подготовка
- trial — Испытание
- crisis — Узкое место
- result — Результат
- integration — Интеграция

Важно: это не игра ради игры. Это слой смысла, который помогает видеть движение по важным проектам.

Возможные элементы:

- карта пути проекта;
- этап текущего фокуса;
- небольшие achievement markers;
- «старец/проводник» как визуальный символ, но не как главный UI;
- опыт/уровни только если они основаны на реальных завершенных действиях.

## Три дизайн-концепта для прототипирования

### Concept A — Dense Tracker

Референсы: Linear / Weeek.

Подходит для задач, статусов, недельного планирования.

Структура:

- левое меню;
- верхний command/search;
- центр: плотные списки/таблицы;
- правый inspector выбранного объекта;
- минимум карточек.

Плюсы: практично, быстро, мало воды.
Минусы: может быть слишком «рабочим» и не давать ощущения личной системы.

### Concept B — Calm Command Center

Референсы: Notion minimal / Readwise / Cron.

Подходит для личной системы дня и рефлексии.

Структура:

- спокойная светлая основа;
- текстовая иерархия;
- рабочие секции без лишних рамок;
- Today как главный экран;
- AI review как компактный список решений.

Плюсы: приятнее для ежедневного использования.
Минусы: риск стать слишком «заметочным», если мало интерактивности.

### Concept C — Hero Projects

Референсы: productivity tracker + легкая RPG-карта.

Подходит для жизненных проектов и больших фокусов.

Структура:

- проекты как маршруты;
- этапы hero journey;
- задачи привязаны к этапам;
- прогресс визуальный, но не декоративный;
- достижения только для значимых переходов.

Плюсы: эмоционально цепляет, помогает видеть смысл.
Минусы: нельзя делать главным интерфейсом для всех задач — перегрузит.

## Рекомендация по направлению

Не выбирать один стиль целиком. Лучший MVP:

- базовый интерфейс: Concept A + B;
- Projects screen: аккуратно добавить Concept C;
- геймификацию оставить на уровне больших проектов, не ежедневных задач.

## Как использовать Open Design

Open Design можно использовать как инструмент генерации и сравнения дизайн-направлений, но не как источник финального решения.

Правильный промпт для Open Design / Codex / Claude:

> Create 3 static HTML concept screens for a PWA Second Brain / Daily OS. Do not copy references. Use them only as design grammar: Linear density, Notion calm typography, Readwise reading clarity, Cron clean scheduling. The product is operational: inbox processing, today plan, projects with hero journey stages, and AI audit review. Avoid generic dashboard widgets and fake analytics. Every block must support an action, decision, comparison, or editable object.
